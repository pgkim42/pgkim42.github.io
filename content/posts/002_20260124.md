## 배경

면접에서 이런 질문을 받았다.

> "commit과 push의 차이점을 설명해주세요."

순간 "음..." 하고 말이 멈췄다.

매일 그냥 자연스럽게 쓰던 명령어를 정확한 용어로 설명을 하려니 쉽게 입이 떨어지지 않았다. 잠시 양해의 시간을 구하고 머릿속으로 정리해서 대답을 하긴 했지만, 뭔가 만족스러운 대답은 아니었다.

면접을 다 마치고 나오는 길에도 다른 질문은 생각나지 않고, git 관련 질문이 머릿속에 오래 남았다. 평소에 당연하게 사용하던 것을 남에게 왜 쓰는지 설명할 수 없을 정도로 내가 정리를 전혀 안 했다는 게 느껴졌다.

---

## commit과 push

### commit은 내 컴퓨터에 저장

commit은 내 로컬 저장소에 현재 상태를 저장하는 거다.

```
Working Directory → (add) → Staging → (commit) → Local Repository
```

파일 수정하고, `git add` 하고, `git commit` 하면 로컬에 스냅샷이 생긴다. 이 시점까지는 내 컴퓨터 안에서 일어나는 일이다.

### push는 원격으로 보내기

push는 로컬의 커밋들을 GitHub 같은 원격 저장소로 보내는 거다.

```
Local Repository → (push) → Remote Repository
```

그래서 혼자 개발할 때는 commit만 해도 된다. 하지만 팀과 일하거나, 백업을 원하면 push를 해야 한다.

| 개념   | 뭐냐면               |
| ------ | -------------------- |
| commit | 내 컴퓨터에 저장     |
| push   | 원격 저장소에 업로드 |

## branch는 뭐냐

branch는 독립된 작업 공간이다.

메인 코드(main)를 건드리지 않고, 새 기능을 시도해볼 수 있는 실험실 같은 거다. 실패하면 그냥 브랜치 지우면 끝. 메인은 멀쩡하다.

```bash
git checkout -b feature-new-ui
```

이 명령 하나로 새로운 우주가 하나 생긴다. 거기서 막 고치고 깽판 치다가, 되면 main에 합치고 안 되면 날리면 된다.

> 사실 branch가 없는 버전 관리는 상상하기도 싫다. 한 줄로 쭉 이어지는 역사에서 실패를 감내할 수가 없으니까.

---

## merge vs rebase

둘 다 브랜치 합치는 건데, 방식이 다르다.

### merge는 역사를 그대로 합침

```
A---B---C    main
     \
      D---E  feature
```

merge 하면:

```
A---B---C---F    main
     \   /
      D---E  feature
```

새로운 커밋 F가 생기고, 두 갈래가 합쳐졌다는 게 기록으로 남는다. 뭘 어떻게 합쳤는지 역사가 다 보인다.

### rebase는 역사를 펴서 합침

```
A---B---C    main
     \
      D---E  feature
```

rebase 하면:

```
A---B---C---D'---E'  feature
```

내 커밋 D, E를 main의 끝으로 "다시 붙인다". 결과적으로 역사가 한 줄로 쭉 이어진다. 깔끔해 보인다.

### 뭐를 쓰면 되냐

- 팀 브랜치를 합칠 때: merge가 안전하다. 역사가 보존되니까.
- 내 로컬 브랜치 정리할 때: rebase가 깔끔하다.

나는 merge를 더 많이 사용했다. 커밋 기록이 갑자기 줄어들어서 곤란해졌던 상황이 있어서 그런가, 커밋 기록을 항상 남겨놔야했다. 코딩 에이전트를 사용하고 나서부턴 더 잘게 쪼개서 커밋을 해서 지저분해 보일 수도 있지만, 커밋 기록을 나보다 더 빠르고 정확하게 훑어주는 도구가 있으니 별 걱정은 없었다.

## stash

stash는 커밋 안 한 변경사항을 잠깐 치워두는 거다.

작업 중인데 갑자기 다른 브랜치로 이동해야 할 때가 있다. 근데 변경사항이 있으면 checkout이 안 된다.

```bash
git stash        # 치워두기
git checkout main
git stash pop    # 다시 꺼내기
```

이거 모르면 일단 커밋하고 이동해야 하는데, "WIP: 작업 중" 같은 의미 없는 커밋이 발생하게 된다.

## worktree

worktree는 같은 저장소를 여러 디렉토리에서 동시에 작업할 수 있게 해준다.

```bash
git worktree add ../project-main main
git worktree add ../project-hotfix hotfix-branch
```

이제 한 터미널에서는 `project-main`에서 작업하고, 다른 터미널에서는 `project-hotfix`에서 작업할 수 있다. 브랜치 전환할 필요가 없다.

## 정리

| 개념     | 핵심                        |
| -------- | --------------------------- |
| commit   | 로컬 스냅샷                 |
| push     | 원격 업로드                 |
| branch   | 독립 작업 공간              |
| merge    | 역사 보존하며 합치기        |
| rebase   | 역사를 선형으로 정리        |
| stash    | 임시 보관함                 |
| worktree | 여러 디렉토리에서 동시 작업 |
